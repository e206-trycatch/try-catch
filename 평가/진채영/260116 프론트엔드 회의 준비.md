### 멀티 모드(Post-MVP) 관련

- 기능 명세서에 따르면 `멀티 대기방`, `실시간 동기화` 등은 대부분 **Post-MVP**입니다.
- 초기 개발 단계에서는 **Member C**가 싱글 모드 게임 로직을 완성하는 데 집중하고, 여유가 생기면 **Member B**가 멀티 로비 UI를, **Member C**가 웹소켓 연동을 준비하는 방식으로 확장하세요.

### 공통 컴포넌트 전략 (Atomic Design 응용)

- 현재 구현할 모달창들이 많음 & 방탈출 컨셉 고려 → 컴포넌트 구조화해야함
- **모달 시스템**
    - 모달이 페이지마다 제각각 렌더링되면 관리 힘들어짐.
        
        ⇒ `Global Modal Manager` 도입
        
        - **구조:** `App.tsx` 최상단에 `<GlobalModal />` 하나만 두고, 전역 상태(Zustand, Recoil 등)로 모달의 `type`과 `props`를 제어합니다.
        - **담당:** **Member B** (디자인 총괄이 모달의 통일성을 잡아야 함)
        - **예시 컴포넌트:**
            - `BaseModal`: 배경(Dimmed), 닫기 버튼, 테두리 스타일 등을 가진 껍데기.
            - `AlertModal`: 단순 메시지 확인용.
            - `ConfirmModal`: 확인/취소 버튼이 있는 모달.
            - `ContentModal`: 내부에 복잡한 Form이나 게임 설정이 들어가는 모달.
- **UI 컴포넌트 (Design System)**
    - 방탈출 컨셉(어두운 배경, 네온, 픽셀 아트 등)에 맞춰 재사용성을 높여야 gka
        - **Button:** `Primary`(게임 시작), `Secondary`(취소), `Ghost`(텍스트 링크), `IconButton`(설정 아이콘)
        - **Input/Textarea:** 로그인, 채팅, 코드 입력(에디터 아님) 등에 쓰일 스타일.
        - **Badge/Tag:** 난이도(Easy/Hard), 언어(JS/Python), 상태(성공/실패) 표시.
        - **Card:** 테마 선택, 랭킹 리스트 등에 쓰일 박스 형태.
        - **LoadingSpinner:** 채점 중, 페이지 로딩 시 사용할 커스텀 로더 (방탈출 컨셉에 맞게 열쇠가 돌아간다거나 하는 등).
- **기능성 컴포넌트 (Logic)**
    - **CodeEditor:** Monaco Editor를 래핑한 컴포넌트. (Member C 담당)
    - **Timer:** 남은 시간을 보여주는 컴포넌트. (Member C 담당)
    - **Terminal/Console:** 코드 실행 결과나 에러 로그를 보여주는 텍스트 박스.
- Directory 구조 예시 (React.ts)

```
src/
├── components/
│   ├── common/          # (Member B 주도)
│   │   ├── Button.tsx
│   │   ├── Modal/       # BaseModal, ModalManager
│   │   ├── Badge.tsx
│   │   └── Layout.tsx   # 헤더, 배경 등
│   ├── game/            # (Member C 주도)
│   │   ├── Editor.tsx
│   │   ├── Terminal.tsx
│   │   └── Timer.tsx
│   └── user/            # (Member A 주도)
│       └── ProfileCard.tsx
├── pages/
│   ├── Login/
│   ├── Lobby/           # 테마 지도 (Member B)
│   ├── GameSetting/     # (Member C)
│   ├── Game/            # (Member C)
│   └── Result/          # (Member A)
├── store/               # 전역 상태 (모달 상태, 유저 정보, 게임 설정값)
└── styles/              # 글로벌 테마, 컬러 팔레트
```

### 협업을 위한 제언

1. **Member B(디자인) & C(게임)의 접점:**
    - **"게임 설정 모달/페이지"** 가 접점.
    - B가 모달의 껍데기(디자인)를 만들면, 
    C가 그 안에 언어 선택 로직과 Form 데이터를 넣는 식으로 작업 진행
2. **Member A(결과) & C(게임)의 접점:**
    - 게임이 끝나고 **"결과 페이지"** 로 넘어갈 때 어떤 데이터를 넘겨줄지(채점 결과, 소요 시간, 코드 내용 등) 미리 API 명세나 전역 상태 구조를 합의해야 합니다.
3. **우선순위:**
    - 가장 먼저 **Member B**가 `Layout`과 `ModalManager`를 잡아줘야 A와 C가 작업을 시작할 때 UI 통일성이 깨지지 않음

### 1. 데이터 흐름의 '바통 터치' 구간 정의하기 (가장 중요)

역할이 나누어지는 경계선에서 데이터가 유실되거나 꼬일 위험이 큽니다. 특히 다음 두 구간의 데이터 명세를 미리 정해야 합니다.

- **Lobby(B) → Game(C) 진입 시:**
    - 단순히 페이지 이동만 하면 안 됩니다. 사용자가 선택한 **`themeId`(테마), `language`(언어), `difficulty`(난이도)** 등의 정보가 게임 페이지로 정확히 넘어가야 합니다.
    - *제안:* URL 파라미터(예: `/game/theme-1?lang=js`)나 전역 상태(Store) 중 무엇을 쓸지 초반에 정하세요.
- **Game(C) → Result(A) 이동 시:**
    - 게임이 끝나면 즉시 결과 페이지로 튕길지, 모달을 띄울지 결정해야 합니다.
    - 결과 페이지에서 보여줄 **`score`(점수), `time`(소요 시간), `pass/fail`(성공 여부), `code`(제출한 코드)** 데이터를 어떻게 넘겨줄지 합의가 필요합니다. (보통은 서버에서 `submissionId`를 받아 결과 페이지에서 다시 조회하는 방식을 씁니다.)

### 2. '채점 로딩(Loading)'의 애매한 소유권 해결

기능 명세서에 따르면 **"채점 진행률 표시"** 와 **"TMI/지식 문구 표시"** 기능이 있습니다. 이 부분은 Member A(결과 화면)와 Member C(게임 화면) 사이에 걸쳐 있습니다.

- **상황:** 사용자가 게임 화면(C)에서 '제출' 버튼을 누르면 -> 로딩 화면이 뜨고 -> 결과 화면(A)으로 갑니다.
- **보완점:**
    - 이 '채점 중' 상태를 **별도의 페이지**로 만들지, 아니면 게임 화면 위의 **Overlay(덮어씌우기) 컴포넌트**로 할지 정해야 합니다.
    - 사용자 경험상 Overlay가 자연스럽습니다. Member A가 **"채점 로딩 컴포넌트"** 를 예쁘게 만들어주면, Member C가 그걸 가져다가 제출 버튼 클릭 시 띄우는 방식으로 협업하세요.

### 3. Member B: '반응형' 포기 여부 결정 (Design)

'방탈출 + 코드 에디터'는 작은 화면에서 구현하기 매우 어렵습니다.

- **현실적인 조언:** 모바일 웹까지 완벽하게 지원하려고 하면 에디터 UI가 망가질 확률이 높습니다.
- **전략:** 과감하게 **"데스크탑/태블릿 전용"** 서비스로 스펙을 좁히거나, 모바일에서는 "게임 플레이 불가능, 랭킹 확인만 가능"하도록 제한하는 것을 추천합니다. 이를 Member B가 디자인 단계에서 확정 짓고 가이드해야 C가 CSS로 고생하지 않습니다.

### 4. Member C: 에디터와 레이아웃 (Technical)

와이어프레임을 보면 코드 에디터와 문제 설명 창이 나뉘어 있습니다. 개발자들이 가장 많이 쓰는 기능인 **"창 크기 조절(Resizable Split Pane)"** 기능이 없으면 매우 답답할 수 있습니다.

- **기술 검토:** 단순히 `div`를 반반 나누는 것보다, `react-resizable` 같은 라이브러리를 사용하여 창 크기를 조절할 수 있게 만들면 완성도가 훨씬 높아 보입니다. 이 부분은 C가 기술 스택을 미리 검토해야 합니다.

### 5. API가 나오기 전 'Mocking' 합의

백엔드 개발 속도가 프론트엔드와 딱 맞지 않을 수 있습니다. 3명이 동시에 작업하려면 **API가 없어도 화면이 나와야 합니다.**

- **데이터 형식 통일:**
    - "로그인 성공 시 응답 JSON은 이렇게 올 것이다"
    - "문제 조회 시 JSON은 `{ problemId: 1, content: "..." }` 형태다"
    - 이런 약속을 `Notion`이나 `Swagger`에 미리 적어두고, 각자 **더미 데이터(Mock Data)** 를 만들어 작업하세요. 그래야 A가 로그인 만드는 동안 C가 게임 로직을 짤 수 있습니다.
