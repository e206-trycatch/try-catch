## 1. 코드 채점 및 디버깅 엔진

사용자가 제출한 코드를 검증하고 피드백을 주는 시스템의 구축하기 위해,
아래의 2가지 방법을 고민하며, 관련 내용을 학습했습니다.

### Docker Sandbox를 통한 보안 및 격리 환경 학습

* **컨테이너 기반 격리 원리**: 호스트 OS와 분리된 독립된 환경에서 신뢰할 수 없는 코드를 실행하는 메커니즘을 이해했습니다.
* **리소스 제한 설정**: `cgroups`를 활용하여 CPU 사용량, 메모리 제한, 프로세스 개수 제한을 설정하여 시스템 가용성을 확보하는 방법을 배웠습니다.
* **보안 취약점 방어**: 샌드박스 내부에서의 네트워크 통신 차단, 파일 시스템 쓰기 권한 제한 등을 통해 쉘 인젝션이나 악성 스크립트 실행을 방지하는 보안 전략을 학습했습니다.

* **시행착오 및 기술적 반려 사유**:
    * **프레임워크별 CORS 및 통신 충돌**: 샌드박스 내에서 실행되는 다양한 프레임워크(예: React, Spring)와 외부 채점 서버 간의 통신 시, 각 프레임워크의 기본 CORS 정책이 상이하여 발생하는 충돌 문제를 경험했습니다. 이를 통해 단순한 컨테이너 격리를 넘어, 네트워크 브릿지 설정과 API 게이트웨이 단계에서의 CORS 허용 정책을 정교하게 설계해야 함을 학습했습니다.
    * **빌드 단계의 메모리 관리 병목**: 실행 시점의 메모리 제한뿐만 아니라, 사용자 코드를 '빌드'하는 과정(컴파일, 의존성 설치 등)에서 발생하는 급격한 메모리 점유율 상승 문제를 겪었습니다. 빌드 단계에서 메모리 관리가 실패할 경우 컨테이너가 강제 종료되는 현상이 발생하였는데, 이를 분석하며, 빌드와 실행의 자원 할당량을 분리하거나 빌드 캐시를 최적화하는 전략의 중요성을 깨달았습니다.
    * **특정 라이브러리 의존성 문제**: 모든 라이브러리를 샌드박스 내에서 실시간으로 설치하는 방식은 네트워크 부하와 빌드 시간 초과를 야기함을 확인했습니다. 이로 인해 '실시간 설치' 방식 대신 '사전 빌드된 기본 이미지' 방식을 학습하였습니다.

### AI 기반 코드 피드백 시스템

* **프롬프트 엔지니어링**: LLM에게 코드 문맥, 에러 로그, 테스트 케이스 정보를 구조화하여 전달하고, 유저에게 정답이 아닌 '논리적 힌트'를 주도록 페르소나를 설계하는 기법을 학습했습니다.
* **JSON 파싱 및 데이터 구조화**: AI의 답변을 정형화된 JSON 형태로 받아 서비스 UI에 맞게 가공하고 출력하는 과정을 익혔습니다.

* **AI 피드백 전략의 장점 (Learning Strengths)**:
    * **유연한 논리 분석**: 전통적인 정적 분석 도구가 잡아내지 못하는 코드의 '의도'와 '논리적 흐름'을 AI가 파악하여, 유저가 왜 이런 실수를 했는지 맥락에 맞는 피드백이 가능함을 확인했습니다.
    * **구현 생산성**: 복잡한 규칙 기반 엔진을 직접 개발하지 않고도, 고도화된 프롬프트 설계만으로 다국어(Python, JS, Java 등) 대응이 가능한 피드백 시스템을 빠르게 구축할 수 있는 효율성을 학습했습니다.
    * **개인화된 학습 가이드**: 정답 코드만 제시하는 것이 아니라 유저의 현재 수준에 맞춰 단계별 힌트를 제공함으로써, 학습 효과를 극대화하는 교육적 설계 방식을 구현할 수 있다는 부분이 매력적이였습니다.

* **기술적 제약 및 단점 (Challenges & Mitigation)**:
    * **할루시네이션 리스크**: AI가 존재하지 않는 문법 오류를 지적하거나 잘못된 로직을 제안할 가능성을 인지했습니다. 이를 방지하기 위해 샌드박스의 실제 실행 결과를 AI에게 교차 검증용 컨텍스트로 제공하는 전략을 학습했습니다.
    * **비용 및 토큰 최적화**: API 호출마다 발생하는 비용 문제를 해결하기 위해, 코드 전체가 아닌 변경된 부분이나 핵심 로직만 추출하여 전달하는 방식의 중요성을 깨달았습니다.
    * **응답 지연(Latency)**: 실시간 피드백을 원하는 유저에게 AI의 생성 시간은 병목이 될 수 있음을 확인했습니다. 이를 해결하기 위해 비동기 처리나 스트리밍 방식의 UI 반영 필요성을 검토했습니다.

###  AI 피드백 시스템 고도화를 위한 기술 검토 및 분석

단순한 프롬프트 주입의 한계를 극복하고, 피드백의 정확도와 신뢰성을 높이기 위해 다음 세 가지 최적화 전략의 도입 가능성과 장단점을 분석했습니다.

#### 1. 파인튜닝 (Fine-tuning): 특정 도메인 최적화
* **검토 내용**: 서비스만의 고유한 피드백 스타일(예: 친절한 튜터 모드)과 특정 프로그래밍 언어의 디버깅 패턴을 모델에게 직접 학습시키는 방식입니다.
* **구현 프로세스**: 
    1. 고품질의 [코드-에러 로그-교정 힌트] 데이터셋 수집 및 정제.
    2. JSONL 등 모델이 학습 가능한 형태로 데이터 포맷팅.
    3. 사전 학습된 모델(LLM)에 해당 데이터를 학습시켜 가중치 업데이트.
* **분석 결과**:
    * **장점**: 모델이 서비스의 페르소나를 완벽히 이해하며, 프롬프트 길이를 줄여도 일관된 출력 형식을 유지할 수 있습니다.
    * **단점**: 양질의 데이터셋 구축에 막대한 시간과 비용이 소요되며, 새로운 언어나 기술 스택이 추가될 때마다 재학습이 필요하다는 유연성 부족의 우려가 있습니다.

#### 2. RAG (Retrieval-Augmented Generation): 외부 지식 참조
* **검토 내용**: AI가 학습하지 못한 최신 라이브러리 정보나 내부 모범 사례(Best Practice)를 외부 데이터베이스에서 검색하여 답변에 포함하는 방식입니다.
* **구현 프로세스**:
    1. 신뢰할 수 있는 기술 문서나 정답 해설지를 벡터화(Embedding)하여 Vector DB에 저장.
    2. 사용자가 코드를 제출하면 질문과 유사한 문서를 DB에서 검색(Retrieval).
    3. 검색된 내용을 프롬프트에 컨텍스트로 포함하여 답변 생성(Augmentation).
* **분석 결과**:
    * **장점**: 할루시네이션(환각 현상)을 획기적으로 줄일 수 있으며, 실시간으로 변하는 기술 트렌드를 재학습 없이 반영할 수 있어 신뢰도가 높습니다.
    * **단점**: Vector DB 구축 및 임베딩 모델 관리 등 시스템 복잡도가 증가하며, 검색 단계로 인한 응답 지연이 발생할 수 있습니다.



#### 3. Hyperparameter (Temperature) 조정: 출력 제어
* **검토 내용**: 모델의 확률적 분포를 조절하여 답변의 창의성이나 일관성을 제어하는 매개변수를 최적화하는 방식입니다.
* **구현 프로세스**: API 호출 시 `temperature` 값을 0.0에서 1.0 사이로 조정하며 테스트 진행.
* **분석 결과**:
    * **Low Temp (0.1~0.2)**: 코드 문법 오류 분석 등 정답이 정해진 영역에 적합하며, 동일 질문에 대해 일관된 답변을 보장합니다.
    * **High Temp (0.7~0.9)**: 다양한 풀이 방식 제안 등 창의적인 힌트가 필요할 때 유용하지만, 답변이 산만해질 위험이 있습니다.
    * **결론**: 채점 시스템의 특성상 낮은 수치를 유지하여 논리적 정확성을 우선시하는 방향이 적합함을 확인했습니다.

## 2. CI/CD 및 인프라 (DevOps)

###  버전 관리 및 배포 도구 분석
* **GitLab vs GitHub**: 
    * GitHub의 오픈 소스 생태계와 Actions 기반의 확장성을 학습했습니다.
    * GitLab의 자체 호스팅 기능과 내장된 CI/CD 파이프라인(GitLab Runner)의 통합 관리 장점을 비교 분석했습니다.
* **GitLab + Jenkins + AWS 파이프라인 연동**:
    * **Webhook**: GitLab의 특정 이벤트(Push, Merge) 발생 시 Jenkins를 호출하는 트리거 메커니즘을 이해했습니다.
    * **Jenkins Pipeline**: `Jenkinsfile` 스크립트를 통해 빌드-테스트-이미지 빌드-푸시로 이어지는 자동화 워크플로우(Pipeline as Code)를 구축하는 법을 배웠습니다.
    * **AWS 배포**: ECR(Container Registry)에 저장된 이미지를 EC2 서버에서 풀링하여 컨테이너를 교체하는 지속적 배포(CD) 과정을 학습했습니다.

## 3. 실시간 통신 및 네트워크 프로토콜

###  WebRTC (Peer-to-Peer)
* **P2P 통신 원리**: 서버 부하를 줄이고 지연 시간을 최소화하기 위해 브라우저 간 직접 데이터를 송수신하는 메커니즘을 이해했습니다.
* **NAT Traversal**: 사설 IP 환경에서 통신을 가능하게 하는 STUN 서버와 TURN 서버의 필요성 및 ICE Candidate 교환 과정을 학습했습니다.
* **Signaling**: 연결 수립을 위해 필요한 SDP(Session Description Protocol) 교환을 위한 시그널링 서버 설계법을 익혔습니다.

###  Web Socket
* **양방향 통신**: HTTP의 단방향 통신 한계를 극복하고 연결을 유지하는 핸드셰이크 과정을 학습했습니다.
* **상태 관리**: 실시간 채팅, 동시 편집 알림 등 즉각적인 데이터 동기화가 필요한 기능에 WebSocket을 적용하는 법을 배웠습니다.
* **비교 학습**: 데이터 전송 지연이 극도로 낮아야 하는 스트리밍(WebRTC)과 안정적인 메시지 전달이 중요한 통신(WebSocket)의 차이점과 적합한 유즈케이스를 구분했습니다.

## 4. 협업 프로세스 및 기획 설계

###  Git 전략 및 컨벤션
* **Git Flow**: `Master`, `Develop`, `Feature`, `Hotfix` 브랜치 전략을 통해 팀 단위 개발 시 코드 충돌을 최소화하고 릴리즈 관리를 체계화하는 법을 익혔습니다.
* **Message Convention**: 협업 효율을 위해 커밋 메시지의 접두사(`feat`, `fix`, `refactor` 등)를 정의하고 일관성 있게 관리하는 습관을 길렀습니다.

###  Jira 기반 애자일 관리
* **이슈 트래킹**: 요구사항을 에픽(Epic) - 스토리(Story) - 태스크(Task) 단위로 세분화하여 백로그를 관리하는 법을 배웠습니다.
* **스프린트 운영**: 정해진 기간 내 업무 범위를 설정하고, 시각화 도구를 활용하여 팀의 작업 진행 경과를 확인하는 법을 익혔습니다.

###  Figma를 활용한 와이어프레임 설계
* **UI/UX 설계 원칙**: 사용자 경험을 고려하여 직관적인 인터페이스를 설계하는 법을 학습했습니다.
* **프로토타이핑**: 실제 개발 전 화면 간 인터랙션을 정의하여 기획의 논리적 허점을 미리 발견하고 수정하는 과정을 거쳤습니다.
* **디자인 시스템**: 컴포넌트 재사용성을 고려한 에셋 관리 기법을 익혀 개발 단계에서의 생산성을 높였습니다.
