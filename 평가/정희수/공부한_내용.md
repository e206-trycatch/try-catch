# 공부한 내용

## 기획 확정 후 흐름 정리 및 기술 분석

1. 제출 기반 팀배틀 게임 흐름
    
    기능
    
- 팀이 제출(코드 diff 또는 수정 파일)을 하면 서버가 채점
- 채점 결과(PASS/FAIL)와 점수, 로그 요약을 즉시(또는 대기 후) 반환
- 팀 룸에는 실시간으로 “제출 접수/채점중/완료” 상태가 뜸

기술적 구현

- 제출 이벤트 상태 머신:
    - SUBMITTED → QUEUED → RUNNING → PASS/FAIL → SCORE_UPDATED
- 채점은 동기 처리 금지
    - API 서버는 “접수”만 하고 즉시 submissionId 반환
    - 워커가 큐에서 가져가서 채점 후 결과 저장
- 실시간 전파
    - WebSocket로 룸에 상태 브로드캐스트

2-0) 명세서 제공

유저에게 API 명세서를 제공하고 디버깅을 진행한다.

### A. 팀은 나눠서 편집 (권장 MVP)

각자 다른 파일 담당

예:

- A: auth.ts
- B: Login.tsx
- C: userStore.ts

→ 제출은 한 사람이 묶어서

기술 난이도: **하**

---

### B. 파일 잠금(lock) 방식

한 사람이 파일을 “잡으면” 다른 사람은 읽기 전용

예:

- A가 Login.tsx 열면
- B, C는 “편집 중” 표시만 보고 못 고침

기술:

- WebSocket + Redis Lock
- Monaco Editor read-only 토글

난이도: **중**

---

### C. 실시간 공동 편집 (Google Docs 스타일)

CRDT/OT 적용

기술:

- Yjs / Automerge + WebSocket
- Monaco Editor binding
- 상태 복구, 연결 끊김 처리

난이도: **최상**

2-2) 제출(Submission) 수집/검증

기능

- 제출물 포맷: “허용 파일만” 수정 가능
- 변경량 제한, 금지 파일 수정 시 즉시 실패 처리
- 제출 쿨다운/레이트리밋(난사 방지)

기술적 구현

- 제출물 포맷 추천(치팅/운영 안정성)
    1. allowed_files 기반 “수정 파일 전체 내용” 제출 (구현 쉬움)
    2. unified diff 제출 (저장 효율 좋고 변경량 계산 쉬움)
- 서버 검증(필수)
    - 허용 파일 목록 외 수정 여부 검사
    - 변경 라인 수 상한(예: 200~500 라인)
    - 금지 파일: package.json, lockfile, build.gradle, settings.gradle, pom.xml, 스크립트 파일 등
- 레이트리밋/쿨다운
    - 유저당/팀당 TTL 카운터(예: 15~30초)
    - 동시 제출 제한(팀 룸 당 1개 RUNNING만 허용 등)

3) 채점 파이프라인(React + Spring Boot “통합 빌드/테스트”)

기능

- 제출마다 동일한 기준으로 빌드/테스트해 PASS/FAIL 판정
- 에러를 “초보자 친화” 형태로 요약 제공
- 팀배틀에서는 빠른 피드백이 중요하므로 “단계형 파이프라인”이 필요

기술적 구현(권장: 단계형, 빠른 실패)

0단계: Pre-check (빠른 컷)

- 허용 파일/변경량/금지 키워드 간단 검사
- 여기서 탈락시키면 서버 자원을 거의 안 씀

1단계: 프론트 정적 검사

- eslint(react-hooks 포함) + TypeScript(tsc 사용 시)
- 목적: Hook 규칙 위반, import 오류, 타입 오류를 “빨리” 잡음

2단계: 프론트 테스트

- vitest/jest 중심의 unit/integration 테스트
- (MVP에서 매 제출마다 vite build까지는 비추천: 느림)
- 필요하면 난이도 높은 스테이지만 build를 켬

3단계: 백엔드 테스트

- gradle test 또는 mvn test
- 스테이지 요구사항을 테스트로 강제(예: 예외 처리/Validation/트랜잭션)

4단계: 통합/E2E

- Playwright 등으로 “프론트+백” 통합 동작 검증
- 팀배틀 “매 제출마다”는 비용이 큼 → 결승/최종 제출(라운드 종료 시)만 적용하는 식이 현실적

결과 표준화

- FAIL_CATEGORY: PRECHECK / LINT / TYPE / FE_TEST / BE_TEST / BUILD / TIMEOUT / SANDBOX
- FAIL_SUMMARY: 초보자도 이해 가능한 2~3줄 요약
- DETAIL_LOG: 원문 전체는 길고 위험(정보 노출)하니, 상위 N줄만

4) 서버 샌드박스(악성 코드/DoS/유출 방어) — 필수

기능

- 유저 제출로 서버가 털리거나 죽지 않게 “격리 실행”
- 무한 루프/메모리 폭주/프로세스 폭주를 제한

기술적 구현

- 컨테이너 격리(권장)
    - 채점 워커는 “컨테이너 내부”에서만 빌드/테스트 실행
    - CPU/메모리 제한
    - 프로세스 수 제한
    - 파일시스템: 작업 디렉토리만 쓰기 허용(가능하면 나머지 read-only)
    - 외부 네트워크 차단(기본 deny)
- 타임아웃
    - 전체 채점 타임아웃(예: 30초~60초)
    - 단계별 타임아웃(예: eslint 5초, test 20초 등)
- 동시성 제한
    - 워커 동시 실행 2~3개 고정(서버가 죽는 걸 방지)
- 로그 폭주 제한
    - 로그 수집 라인 수/크기 상한

“허용 파일 제한”은 샌드박스만큼 중요합니다.

- 빌드 스크립트/의존성 파일 수정 가능하게 열어두면 공격 표면이 급증합니다.

────────────────────────────────────────

5) 팀배틀 룸/실시간 상태/스코어보드

기능

- 방 생성/참가/시작/종료
- 현재 라운드(스테이지), 제한시간, 제출 상태 표시
- 누가 제출했는지, 채점 중인지, 통과했는지 실시간 반영

기술적 구현

- 룸 상태 머신
    - WAITING → RUNNING → FINISHED
    - 라운드별 stage_version 고정(채점 재현성)
- 실시간 전파
    - WebSocket(STOMP) 또는 SSE
    - 이벤트 기반 메시지(전체 상태를 매번 보내지 않음)
        - SUBMISSION_CREATED, GRADING_STARTED, GRADING_FINISHED, SCORE_UPDATED
- 데이터 저장 전략
    - 중간 상태(타이머/현재 제출): 메모리/Redis
    - 결과/전적: DB

────────────────────────────────────────

6) 점수 산정(제출마다 채점한다는 전제)

기능

- “PASS”를 기본으로 하고 시간/시도/힌트/페널티로 순위 결정
- 팀전은 팀 점수 + 개인 기여도 분배(선택)

기술적 구현(팀배틀에 적합한 단순 모델)

- 라운드 점수 = baseScore + timeBonus − attemptPenalty − hintPenalty
- timeBonus는 “라운드 시작~첫 PASS 제출 시점” 기준(팀전은 속도가 핵심)
- attemptPenalty는 팀 단위 제출 횟수로 적용

────────────────────────────────────────

7) 성능/부하(“매 제출마다 빌드/테스트”의 현실적인 운영)

핵심 결론

- 이 구조에서 성능 병목은 100% 채점(빌드/테스트)입니다.
- 동시 유저 수가 아니라 “동시 제출 수”가 서버를 터뜨립니다.

기술적 구현(필수 운영 장치)

- 큐 + 워커 분리(필수)
    - API 서버는 접수만 담당
    - 워커는 빌드/테스트 전담
- 캐시(효과 큼)
    - Node 의존성 캐시(pnpm store 등)
    - Gradle 캐시
    - 스테이지 베이스 워크스페이스 재사용(고급)
- 전략적 파이프라인(매 제출마다 ‘풀빌드’ 하지 않기)
    - 기본은 lint+test 중심
    - build/E2E는 라운드 종료 “최종 제출”에만 수행하는 옵션이 매우 현실적

9) “매 제출마다 빌드”를 그대로 하면 생기는 문제와 현실 해법

문제

- 팀배틀에서 제출이 잦으면 빌드가 비용 폭탄
- t2.xlarge 같은 단일 서버에서 병목이 빨리 발생

현실 해법(게임 룰로 해결 가능)

- 제출마다 “풀빌드”가 아니라 “단계형” or 제출 횟수 제한(시간 제한)
    - 제출 즉시: 빠른 검사(eslint/tsc/unit test)로 즉시 피드백
    - 라운드 종료/최종 제출: build + 통합 테스트로 확정 판정
        
        이렇게 하면 “게임성(즉시 피드백)”과 “정확성(최종 검증)”을 둘 다 잡습니다.
        

if) 빌드를 하지 않을 수 있는 방법
프론트는 프론트에서 실행을 다하고 백엔드에서는 프론트에서 api를 잘 불러왔는지 체크

백에서는 Javaparser를 이용하여 문법, 요소, 무한 루프 여부 등을 체크하여 정답을 측정한다.

- 프론트 코드
    
    → 브라우저(iframe/worker)에서 직접 실행
    
- 서버는 프론트 코드를 실행하지 않는다.
- 프론트는 API를 호출하고,
    
    → 모든 요청은 Back(/proxy)를 통해 나간다.
    
- Back는
    
    → 요청이 명세와 일치했는지(메서드, URL, body, 헤더, 순서 등)를 채점한다.
    
- 백엔드
- 사용자는 Spring 코드(컨트롤러/DTO/서비스)를 고쳐서 제출
- Back은
    
    → JavaParser로 AST를 파싱
    
    → 문제별 규칙(rule)을 검사

## 1. WebSocket 기반 실시간 이벤트 스트림

- WebSocket(STOMP) 기반 방 단위 토픽 구조
- `/topic/room/{roomId}` 형태의 채널 분리
- READY, START, SUBMIT, RESULT, CODE_SAVED, TIMER 등 게임·채점 이벤트 브로드캐스트
- WebSocket = 실시간 데이터 전송 채널이 아닌 **상태 변경 이벤트 버스**
- 모든 클라이언트는 동일한 이벤트 스트림을 수신
- 클라이언트 상태는 이벤트 기반으로 갱신
- 서버가 게임 상태·채점 상태의 단일 진실(Single Source of Truth)

---

## 2. Redis 기반 중앙 상태 저장소

- Redis를 단순 캐시가 아닌 **실시간 상태 관리 레이어**로 사용
- 방 상태(room state) 저장
    - WAITING, READY, RUNNING, FINISHED
- 유저 상태(user state) 저장
    - ONLINE, READY, ROLE, SUBMITTING
- 코드 상태 저장
    - fileKey별 version, hash, latest savedAt
- 채점 상태 저장
    - QUEUED, COMPILING, RUNNING, SUCCESS, FAIL, TIMEOUT
- WebSocket 재접속 시 Redis 기준 상태 재동기화
- 서버 재시작 시에도 방·채점 상태 유지
- 동시 요청 간 충돌 방지용 중앙 상태 기준점 역할

---

## 3. 코드 저장 및 다중 파일 관리

- 코드 파일(.java, .py 등) → UTF-8 텍스트 데이터
- 파일 단위 관리 키 구조
    - `(roomId, fileKey)`
- fileKey 예시
    - `Main.java`, `Solution.py`, `src/App.js`
- 저장 시 동작
    - content hash 계산
    - 동일 hash → 저장 및 알림 스킵
    - 변경 시 version 증가
- baseVersion 기반 optimistic locking
    - 오래된 코드 덮어쓰기 차단
    - 동시 저장 충돌 방지
- Redis에 최신 코드 메타데이터 캐싱
- DB에 최신 코드 또는 제출 스냅샷 영속 저장

---

## 4. 실시간 제출·채점 처리 구조

- 제출 요청 → 채점 파이프라인 진입
- 단계 분리
    - COMPILE
    - RUN(TESTCASE)
    - RESULT(EVAL)
- 테스트케이스 기반 정답 판정
- 성공 / 오답 / 컴파일 에러 / 런타임 에러 / 시간 초과 분기
- Redis에 채점 진행 상태 기록
- WebSocket으로 SUBMIT, JUDGE_START, JUDGE_RESULT 이벤트 전파
- 클라이언트는 이벤트 기반으로 채점 진행 UI 갱신

---

## 5. 동시성 제어 및 악성 제출 방지

- 유저 단위 레이트 리밋
    - 저장, 제출 연타 차단
- content hash 기반 중복 제출·저장 차단
- Redis 기반 방 상태 체크
    - 이미 채점 중이면 중복 제출 차단
- 버전 기반 저장 충돌 방지
- 서버 권위(authoritative server) 구조
    - 클라이언트는 상태 변경 불가
    - 모든 상태 변경은 서버 검증 후 확정
