# 📑 WebSocket & STOMP 정리

## 1. WebSocket 개요

* **정의:** 클라이언트와 서버 간에 지속적인 연결을 유지하는 양방향 통신 프로토콜
* **특징:**
* HTTP와 달리 한 번 연결되면 데이터 전송 시 헤더가 매우 가벼움
* 실시간 채팅, 실시간 주식 차트, 게임 등에 필수적


* **한계:** 텍스트나 바이너리 데이터를 주고받을 뿐, 데이터의 **'형식'**이 정해져 있지 않아 복잡한 애플리케이션 구현 시 제약이 있음

---

## 2. STOMP (Simple Text Oriented Messaging Protocol)

* **정의:** WebSocket 위에서 동작하며, 메시지 전송을 위한 **규격(Format)**을 정의하는 프로토콜
* **역할:**
* 메시지 헤더와 본문을 구분하여 목적지(`Destination`) 정보를 포함함
* 브로커 기반의 **Pub/Sub(발행/구독)** 모델을 가능하게 함


* **장점:**
* 메시지 엔드포인트를 자유롭게 설계 가능
* 권한 처리나 메시지 가공이 용이함



---

## 3. 핵심 동작 원리 (Pub/Sub 모델)

1. **구독(Subscribe):** 클라이언트가 특정 경로(`topic/chat/1`)를 구독함
2. **발행(Publish):** 클라이언트가 특정 경로(`app/chat/1`)로 메시지를 보냄
3. **브로커(Broker):** 서버 내의 브로커가 메시지를 가공하거나 해당 경로를 구독 중인 모든 클라이언트에게 전송

---

## 4. WebSocket vs STOMP 비교

| 구분 | WebSocket | STOMP |
| --- | --- | --- |
| **수준** | Low-level 프로토콜 | High-level (WebSocket 위에서 동작) |
| **메시징 방식** | 원시 텍스트/바이너리 | 헤더 기반의 체계적 메시징 |
| **구조** | 단순 연결 유지 | Pub/Sub 구조 지원 |
| **적합성** | 단순한 일대일 통신 | 복잡한 다대다 통신 (채팅방 등) |

---

## 5. 통신 흐름 요약

* **Handshake:** HTTP 프로토콜을 통해 연결을 시도하고 성공 시 WebSocket으로 업그레이드
* **CONNECT:** STOMP 프로토콜을 이용해 클라이언트와 서버 간 논리적 연결
* **SEND:** 메시지 전송 (보통 `/app` 경로 사용)
* **MESSAGE:** 구독 중인 사용자들에게 브로커가 메시지 전달 (보통 `/topic` 경로 사용)
